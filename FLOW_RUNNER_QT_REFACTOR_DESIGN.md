# Flow Runner 原项目分析与 PySide6 重构设计

## 1. 文档目标

本文梳理 `flow_runner_p1.py`、`flow_runner_p2.py`、`flow_runner_p3.py` 的整体框架、数据结构和运行流程，并定义后续 PySide6 标准化重构的目标架构。

本轮只制定设计，不修改原项目代码。旧配置不由新版直接兼容；重构完成后，再依据新旧模型编写一次性配置转换程序。

## 2. 当前项目状态

当前目录不是 Git 仓库，没有 README、依赖清单和自动化测试配置。主要文件如下：

| 文件 | 规模 | 主要职责 |
|---|---:|---|
| `flow_runner_p1.py` | 约 54 KB | 截图、OCR、图像匹配、录制、回放、步骤执行、配置读写 |
| `flow_runner_p2.py` | 约 23 KB | 单流程循环、顶层任务调度、流程串联、配置数据操作 |
| `flow_runner_p3.py` | 约 104 KB | Tkinter 界面、步骤编辑器、流程树、运行控制和设置 |
| `config/flow_runner.json` | 约 139 KB | 全局设置、流程组、流程和步骤配置 |
| `scripts/` | 多个 JSON/PNG | 录制脚本与图像模板 |

现有配置包含 4 个流程组、99 个流程和 159 个步骤。已有步骤类型包括 OCR 检测、图片检测、等待、程序启动、录制回放、鼠标、滚轮和键盘等。

## 3. 原项目整体框架

### 3.1 Part 1：底层能力与步骤执行

`flow_runner_p1.py` 同时承担基础设施、领域数据和业务执行职责：

1. 初始化文件日志和控制台日志。
2. 检测 `pyautogui`、`pynput`、Pillow、pywin32、OpenCV 等可选依赖。
3. 提供普通截图和 BitBlt 截图。
4. 调用 PaddleOCR-json 或 Tesseract，并进行关键词匹配。
5. 使用 OpenCV 进行模板匹配。
6. 解析识别位置、偏移坐标和绝对坐标，执行点击序列。
7. 通过 `RecorderEngine` 录制鼠标移动、点击和滚轮事件。
8. 通过 `PlaybackEngine` 按时间戳、速度和随机抖动回放脚本。
9. 在 `STEP_DEFAULTS` 中定义 12 种固定步骤字典。
10. 由 `StepExecutor` 根据步骤 `type` 分发具体执行方法。
11. 定义流程、流程组和全局配置默认值。
12. 读取、迁移并保存 `config/flow_runner.json`。
13. 在进程退出时清理 PaddleOCR 子进程。

现有固定步骤为：

- `ocr_click`、`ocr_loop`、`ocr_poll`
- `img_click`、`img_loop`、`img_poll`
- `wait`、`mouse_move`、`scroll`
- `keyboard`、`run_script`、`launch_app`

### 3.2 Part 2：流程执行与任务调度

`flow_runner_p2.py` 在 Part 1 之上增加两级执行器：

- `FlowEngine`：执行单个流程的前置等待、循环次数和步骤序列。
- `FlowRunner`：在后台线程中启动任务，控制暂停、继续和停止，并根据 `next_flow` 串联流程。

当前串联仅允许在同一个流程组内进行。`next_flow` 使用组内数组索引，`-1` 表示结束。流程移动或删除时，需要同步修正其他流程的索引引用；跨组移动会重置串联关系。

Part 2 还直接提供流程组、流程和步骤的增删改、移动、复制函数，供 GUI 操作原始字典。

### 3.3 Part 3：Tkinter GUI

`flow_runner_p3.py` 是单文件 GUI 主程序，主要组成如下：

- 硬编码颜色字典和 Tkinter 控件样式。
- 屏幕区域框选和坐标拾取覆盖层。
- 点击动作序列编辑器。
- 键盘动作序列编辑器。
- 根据步骤类型分支构建的 `StepCard`。
- 顶部启动、暂停、停止和起始流程选择栏。
- 左侧流程组树、流程管理和录制脚本区。
- 中间步骤卡片编辑区。
- 底部运行日志区。
- 流程组和流程的重命名、拖拽、复制、删除及串联设置。
- 全局热键监听和设置窗口。
- UI 关闭时的运行任务与 OCR 进程清理。

### 3.4 原项目启动与执行流程

```text
启动 flow_runner_p3.py
  → 设置 Windows DPI
  → 导入 Part 1 和 Part 2
  → 导入阶段初始化日志并检测依赖
  → 读取并迁移 JSON 配置
  → 构建 Tkinter 主窗口
  → 刷新流程树、脚本列表和热键监听

用户编辑流程
  → GUI 控件直接读写嵌套字典
  → 保存步骤卡片数据
  → JSON 写入磁盘

用户启动任务
  → 创建 FlowRunner
  → 后台线程启动 PaddleOCR
  → 从指定流程组和流程开始
  → FlowEngine 执行前置等待
  → 按 loop_count 循环步骤列表
  → 每一步创建 StepExecutor 并按 type 分发
  → 返回 OK 或 ABORT
  → 流程完成后读取组内 next_flow
  → 继续下一个流程或结束任务
  → 回调 Qt/Tk 主线程更新状态与日志
  → 停止 PaddleOCR 并生成运行统计
```

## 4. 当前结构的主要问题

### 4.1 固定类型导致组合爆炸

OCR 和图片检测分别实现 click、loop、poll 三套步骤。它们的重试、间隔、超时、点击和前置动作逻辑高度相似。每增加一种检测能力，都可能再次复制三套执行代码、默认配置和 GUI 分支。

### 4.2 数据、执行与展示重复描述同一类型

一种步骤至少要同时修改：

- `STEP_DEFAULTS`
- `StepExecutor.exec_step()` 和具体执行方法
- `STEP_LABELS`
- `StepCard._build_body()`
- `StepCard._load()` 与 `StepCard.save()`

这些位置依赖字符串键约定，没有统一配置模型和注册信息，容易出现字段不一致。

### 4.3 模块边界不清晰

Part 1 同时导入 Tkinter、创建日志目录、管理外部进程、定义配置和执行业务；导入模块本身就产生副作用。GUI 直接操作配置字典，执行器也直接读取这些字典，无法独立验证模型。

### 4.4 索引引用不稳定

流程跳转使用数组索引。移动、删除和跨组操作都需要额外修正引用，未来支持跨组跳转、调用与返回后会更加脆弱。

### 4.5 运行语义存在隐含约定

单次未命中、重试耗尽、异常和超时没有统一的结构化结果模型。底层方法内部决定何时终止，策略与路由无法独立组合。

### 4.6 可测试性不足

截图、OCR、输入控制、线程、配置和 GUI 紧密耦合。项目没有自动化测试，真实桌面环境成为大部分逻辑的隐含依赖。

### 4.7 UI 难以持续演进

Tkinter 界面、样式和业务操作集中在大型类中。步骤卡片使用大量分支和硬编码样式，不适合后续依据独立设计规范统一调整。

## 5. 重构目标

1. 使用 PySide6 重构桌面界面。
2. 将流程组改为纯组织目录，执行关系允许跨组。
3. 使用“通用步骤 + 条件路由”替代固定组合类型。
4. 新增步骤只显示“检测、执行、控制”三个入口。
5. 步骤创建后可以切换 OCR、图片等检测能力。
6. 支持条件分支、跨组跳转、流程调用与返回。
7. 支持固定次数和 OCR、图片、时间、变量等动态跳转条件。
8. 分离领域模型、执行器、基础设施和 Qt 展示。
9. 通过能力注册表消除大型类型分发代码。
10. 为截图共享、条件结果传递、重试和并发定义确定语义。
11. 所有视觉样式由 QSS 管理，为后续 `DESIGN.md` 留出独立设计入口。

## 6. 新版核心模型

### 6.1 项目层级

```text
Project
└─ FlowGroup（仅用于分类）
   └─ Workflow
      └─ AutomationStep
```

流程和步骤使用稳定 UUID。移动、重命名或跨组整理不会破坏引用。

### 6.2 通用自动化步骤

```text
AutomationStep
├─ condition：可选条件树
├─ actions：条件成功后执行的主动作列表
├─ condition_policy：检测模式、间隔、次数、超时和轮询钩子
├─ action_policy：动作失败重试策略
└─ routes：最终结果对应的下一目标
```

- 没有条件：直接执行主动作。
- 没有主动作：只检测并路由。
- 同时存在：检测成功后依次执行动作。
- 更换 OCR 或图片只替换条件能力，通用策略和路由继续保留。

### 6.3 条件树

条件支持单一叶子以及 AND、OR、NOT 组合。简单模式只显示单个条件，高级模式开放条件组。

第一阶段条件包括 OCR、图片模板、时间、次数和变量比较；第二阶段增加像素颜色、区域变化、窗口与进程状态。

### 6.4 动作

第一阶段动作包括鼠标点击、移动、滚轮、键盘、组合键、等待、启动程序、运行脚本、录制回放、设置变量和流程控制。

检测策略可以配置 `before_attempt_actions` 和 `after_no_match_actions`。它们复用动作配置，但属于策略钩子，不是新的动作类型；主 `actions` 始终只在条件匹配后执行。

### 6.5 路由结果

步骤只在产生最终结果后匹配一次路由：

- `SUCCESS`：检测命中且主动作完成。
- `NOT_MATCHED`：单次检查正常完成但条件为假。
- `TIMEOUT`：等待模式达到次数或时限仍未命中。
- `FAILURE`：检测或动作异常，且异常重试耗尽。
- `CANCELLED`：用户停止或任务取消。

路由目标可以是下一步骤、任意流程、调用流程、返回调用方或结束任务。

## 7. 条件求值与数据传递

### 7.1 感知快照和缓存

每次条件求值创建逻辑 tick。真正的截图由全局 `PerceptionService` 提供：

- 合并同一目标短时间内的截图请求。
- 同一帧支持多个区域裁剪。
- OCR 按帧、区域、引擎、语言和预处理参数缓存。
- 图片检测共享截图，不同模板分别匹配。
- 动作改变画面后递增场景版本并失效相关缓存。

### 7.2 条件结果上下文

条件返回结构化结果，包含 outcome、文本、坐标、矩形、置信度、诊断产物、能力专属数据和子结果。

本步骤动作和路由可以使用绑定表达式：

```text
$result.primary.position
$result.primary.text
$result.children["ocr_a"].text
$result.children["image_b"].position
```

`primary` 只在以下情况存在：

- 单一叶子条件命中。
- OR 组运行时恰好有一个叶子命中。

AND、NOT 和多个叶子同时命中的 OR 不产生 `primary`。复合条件的叶子必须有唯一别名，并通过 `children` 显式引用。绑定不存在的结果会产生 `BindingError`。

结果上下文默认只在当前步骤有效。跨步骤使用时，必须通过设置变量或导出结果显式保存。

## 8. 重试、未命中与超时

`condition_policy.mode` 明确区分：

- `ONCE`：检查一次，条件为假产生 `NOT_MATCHED`。
- `UNTIL`：持续检测，次数或时限耗尽产生 `TIMEOUT`。

执行顺序固定为：

```text
轮询前动作
  → 获取新快照
  → 求值条件
  → 匹配：执行主动作及其独立重试
  → 未匹配：执行本轮未命中动作
  → 策略允许继续：等待并进入下一 tick
  → 策略耗尽：生成最终结果
  → 根据最终结果匹配一次路由
```

策略只负责如何尝试；路由只负责最终结果往哪里走。“忽略、结束、恢复”等决策由路由表达，不放入重试策略。

## 9. 并发与资源一致性

`ResourceCoordinator` 管理窗口、桌面、鼠标和键盘资源：

- 只读检测可以并发。
- 同一窗口的交互动作默认串行。
- 鼠标和键盘默认是全局独占资源。
- 未指定目标窗口时锁定整个桌面。
- 根据检测坐标执行动作前，检查生成该坐标的 frame 是否仍然有效。
- 快照过期时，在独占锁内重新检测并更新坐标后再执行。
- 交互完成后递增场景版本，使相关感知缓存失效。

这一模型保证多流程共享截图时，不会因为另一流程改变画面而继续使用过期坐标。

## 10. Qt 编辑器设计

### 10.1 主布局

采用“A 为主、C 为辅”：

- 左栏：流程组和流程树。
- 中栏：当前流程的步骤序列。
- 右栏：当前步骤的条件、动作、策略和路由属性。
- 新增步骤只选择检测、执行或控制。
- 复杂规则可以进入引导式编辑器。

第一版不实现任意连线的流程图画布。

### 10.2 UI 与样式边界

- Python 只负责布局、行为、数据绑定和语义属性。
- 不在 Python 中散落颜色、字体或局部 `setStyleSheet()`。
- 视觉属性统一放入 QSS。
- 控件通过 `objectName` 和动态属性暴露语义状态。
- `ThemeManager` 统一加载和刷新 QSS。
- `resources/styles/` 和 `resources/icons/` 为后续 `DESIGN.md` 预留。
- Qt 通过 ViewModel 操作领域模型，不直接修改 JSON 字典。

## 12. 错误处理

领域异常至少包括：

- `ConditionError`
- `ActionError`
- `BindingError`
- `RoutingError`
- `ResourceConflict`
- `ConfigurationError`
- `Cancelled`

底层能力不得私自无限重试。后台执行不得直接操作 Qt 控件，只通过信号发送不可变状态。所有等待、轮询和动作序列必须响应取消令牌。配置保存采用临时文件、校验和原子替换，并保留近期备份。

## 13. 测试策略

- 单元测试条件树、`primary` 规则、绑定、策略状态机和路由。
- 测试同帧共享、缓存键、动作后失效和快照重检。
- 测试只读共享、输入独占、资源竞争和取消。
- 测试跨组跳转、流程调用返回、动态循环和失败恢复。
- 测试配置校验、原子保存、引用断裂和版本升级。
- 使用虚拟截图、假 OCR 和假输入适配器完成确定性测试。
- 使用 Qt 测试验证三栏同步、类型切换、属性编辑和引导添加。
- 在 Windows 上手工验收多显示器、DPI、全局热键、真实 OCR 和游戏窗口截图。

## 14. 分阶段修改方向

### 阶段 1：建立可测试核心

创建依赖清单、包结构、领域模型、配置验证、能力接口和基础测试。先实现纯内存的条件、策略、动作与路由状态机，不接入真实桌面。

### 阶段 2：迁移现有底层能力

将截图、OCR、模板匹配、鼠标键盘、程序启动、录制与回放拆成独立适配器。用能力注册表替代固定类型分支，并通过假适配器与真实适配器的契约测试验证行为。

### 阶段 3：实现调度、感知与资源协调

实现跨组跳转、调用栈、运行变量、全局感知缓存、资源锁、快照校验、暂停和取消。用集成测试覆盖动态循环和并发监控。

### 阶段 4：搭建 PySide6 UI 骨架

建立三栏主窗口、ViewModel、步骤序列、属性编辑器和引导添加器。只完成结构与行为，视觉样式统一从 QSS 加载，等待后续 `DESIGN.md` 决定最终外观。

### 阶段 5：接入运行控制与诊断

接入启动、暂停、停止、热键、日志、统计、条件预览、单步运行和资源冲突提示。确保所有后台状态通过 Qt 信号安全更新。

### 阶段 6：补充第二阶段能力

增加像素、区域变化、窗口、进程状态、窗口动作、并行监控和调试工具。每项能力通过注册接口独立添加，不修改核心状态机。

### 阶段 7：配置转换与实际验收

核心重构稳定后，读取旧配置和新版模型，生成一次性转换工具与新配置。使用用户现有流程做真实回归，不在核心代码中长期保留旧格式分支。

## 15. 完成标准

- 原有主要自动化能力可以由通用模型表达。
- OCR 和图片检测不再各自复制 click、loop、poll 执行框架。
- 跨流程组动态跳转、调用返回和条件循环通过自动化测试。
- 条件结果可以安全绑定到动作和路由。
- 多流程共享感知数据，交互动作不会使用失效坐标。
- UI 不直接操作 JSON，不包含硬编码视觉样式。
- 核心执行器可以在没有 PySide6 和真实桌面的测试环境中运行。
- 配置写入可校验、可恢复，并能报告断裂引用。
- 后续能力可通过注册表添加，而不修改核心步骤分发器。
