# 三列响应式操作区与界面易用性改进设计

## 背景

当前主窗口把项目编辑和运行功能集中在顶部两个 `QToolBar` 中。按钮数量较多，功能与左侧流程树、中间步骤列表、右侧配置区的空间关系不明确；工具栏也不会跟随三列各自的宽度换行。步骤卡片目前只有选中项展开，展开后反而隐藏步骤名称。右侧“启动程序”动作选择脚本或批处理文件时，还需要用户手动填写解释器和参数。

本次设计同时处理用户在确认过程中补充的四项界面问题：文件型动作摘要显示完整路径、动态绑定显示内部英文表达式、界面仍有不必要的英文，以及主窗口宽高不能作为本机偏好自动恢复。

## 目标

1. 把功能控件放到其对应列的内容区下方，并随该列宽度自动换行。
2. 所有步骤卡片默认且始终展开，步骤名称始终位于卡片顶部。
3. 用户为“启动程序”选择 `.py`、`.pyw`、`.bat` 或普通可执行文件时，自动形成可运行配置。
4. 文件型动作摘要只显示有辨识度的目标文件名，不暴露冗长完整路径。
5. 普通界面用中文名称展示结果绑定，内部表达式保持兼容。
6. 清理应用维护的用户可见英文，保留必要缩写和技术名称。
7. 主窗口宽高在本机自动保存并于下次启动恢复。

## 非目标

- 不改变项目 JSON、动作配置模型、绑定表达式语法或执行引擎语义。
- 不把本机窗口宽高写入项目文件，也不改变现有三列宽度随项目保存的机制。
- 不增加可折叠步骤卡片；本次确定为全部始终展开。
- 不翻译文件名、用户输入、项目数据、内部配置键、第三方原始错误详情或 OCR、RGB、ID、JSON、Python 等通用名称和缩写。
- 不进行与本次界面改进无关的主题或业务流程重构。

## 方案比较

### 方案 A：每列独立的响应式操作区（采用）

主窗口继续持有现有 `QAction` 和业务槽函数，在每列内容部件外增加统一的列容器；列容器上部承载原内容，下部承载响应式操作区。操作区用动作绑定按钮和带标签的输入控件组成，通过支持高度随宽度变化的流式布局自动换行。

优点是动作状态和快捷键仍只有一个来源，三个操作区能独立响应各自列宽，职责边界清楚，也便于无显示环境测试。代价是分隔器直接子部件会由原面板变为列容器，需要更新现有布局测试。

### 方案 B：保留顶部工具栏并插入换行

Qt 工具栏可以手动换行，但无法根据三个分栏的宽度分别布局，也无法表达按钮属于哪一列。因此不能满足核心要求。

### 方案 C：把三列改成可停靠窗口

停靠窗口能让每个区域拥有自己的工具栏，但会引入拖出、关闭、停靠状态持久化等额外交互，显著改变当前固定三列工作流，超出本次范围。

## 总体架构

新增两个轻量界面单元：

- `ColumnContainer`：纵向承载一个现有内容部件和一个底部操作区，内容部件保持伸展，操作区按其内容计算高度。
- `ResponsiveControlArea`：接收现有 `QAction`、标签和输入部件。动作统一使用 `QToolButton.setDefaultAction()`，自动同步文字、启用状态和触发信号；布局根据可用宽度排列并换行。

`MainWindow` 仍负责创建动作、连接事件和刷新上下文状态。列容器和操作区只负责呈现，不复制保存、撤销、运行或编辑逻辑。现有 `CompactFlowLayout` 的高度随宽度算法可以复用或抽取；实现必须在分隔器拖动后调用几何更新，使换行和操作区高度立即生效。

顶部 `runtimeToolbar` 和 `projectToolbar` 不再加入主窗口。为兼容测试与内部调用，现有 `QAction`、启动选择框和公开面板属性继续保留。

## 三列控件归属

### 左列：流程树

操作区按“运行”“组与流程”“并行监控”形成可辨识的小组，但所有控件参与同一响应式宽度计算：

- 启动组、启动流程。
- 启动、暂停、停止、录制。
- 新增组、复制流程组。
- 新增流程、复制流程、重命名、流程上移、流程下移、移动到组、删除组/流程。
- 新增并行块、编辑并行块、删除并行块。

### 中列：步骤列表

- 从模板新增步骤、新增步骤、复制步骤、删除步骤。
- 上移步骤、下移步骤。
- 单步运行、预览条件。

### 右列：配置区

- 保存、撤销、设置、诊断。

按钮沿用现有上下文启用规则。列变窄时完整按钮向下一行移动，不截断、不覆盖相邻列；列变宽时自动回收到更少行。操作区不设置固定列数。

## 步骤卡片

`StepCardWidget` 初始化时显示标题和正文，`is_expanded` 为真。标题行固定为“序号 + 步骤名称”，正文依次显示检测、每项执行、策略和路由摘要。

选中状态与展开状态解耦。`StepListPanel._on_current_item()` 只更新卡片的选中属性和发出 `stepSelected`，不再对其他卡片调用收起逻辑。工作流刷新后每个列表项都立即用完整卡片 `sizeHint()` 更新高度；后续选择不得改变卡片是否展开。

## “启动程序”文件选择联动

### 触发与边界

通用 `PathFieldEditor` 增加“用户通过文件对话框选择了文件”的信号。只有 `ModelForm` 正在编辑 `LaunchProcessConfig` 的 `path` 字段时才连接自动配置逻辑；手工输入路径不强制改写其他字段，其他路径字段也不受影响。

自动配置由无界面的纯函数生成，输入为所选文件、当前 Python 路径及系统命令解释器路径，输出为程序路径、自动参数前缀和工作目录，便于独立测试。

### 规则

| 所选文件 | 程序路径 | 自动参数前缀 | 工作目录 |
| --- | --- | --- | --- |
| `.py` | 当前全局 Python 的 `python.exe`（即当前进程解释器） | 所选脚本完整路径 | 所选文件目录 |
| `.pyw` | 当前解释器同目录的 `pythonw.exe`；不存在则回退到当前解释器 | 所选脚本完整路径 | 所选文件目录 |
| `.bat` | `%COMSPEC%` 指向的 `cmd.exe`；无有效值时使用 Windows 系统目录中的 `cmd.exe` | `/c`、所选批处理完整路径 | 所选文件目录 |
| 其他文件 | 所选文件完整路径 | 无 | 所选文件目录 |

自动参数前缀置于用户附加参数之前。表单在本次编辑会话中记录上一次自动生成的前缀；再次选择文件时只替换该前缀，保留其后的用户参数。工作目录为空或仍等于上一次自动值时才自动更新；用户已经手工改为其他目录时不覆盖。若解释器候选文件不存在，则使用明确回退规则，并保持表单可继续手工修改。

参数字段继续使用现有列表数据格式，最终配置仍由能力注册表和 `LaunchProcessConfig` 验证。选择文件本身只修改当前表单并触发现有“待应用”状态，不直接写项目。

## 文件型动作摘要

`action_summary()` 增加统一的文件名提取规则：

- `system.launch` 若为 Python 启动形式，显示脚本参数的文件名；若为 `cmd /c`，显示批处理参数的文件名；否则显示程序路径的文件名。
- `recording.playback` 显示录制文件名。
- 缺少有效路径时保持当前能力名称，不抛出摘要异常。

因此步骤卡片和动作编辑器列表会共享一致结果，例如“启动程序：每日任务.py”，项目内部仍保留完整路径。

## 中文结果绑定

### 展示模型

内部值继续使用 `$result.primary.position`、`$result.children["节点名称"].text` 等表达式。界面通过独立的绑定选项模型把内部值映射为中文标签：

- `$result.primary.position` → “当前步骤检测结果 → 主要结果 → 坐标”。
- `$result.primary.text` → “当前步骤检测结果 → 主要结果 → 识别文字”。
- 子节点表达式 → “<检测能力中文名>「<节点名称>」→ <结果字段中文名>”。

这里的“节点名称”使用条件编辑器已有的 `ConditionNode.id`，不新增领域字段。动作编辑器从当前步骤条件树获得可用节点；条件内容或节点名称变化后刷新选项，但通过内部表达式保持已经选择的值。

普通模式用中文下拉选择结果来源和结果字段。无法映射的旧表达式不会丢失，而是在“自定义表达式”入口显示原值。自定义表达式属于高级操作，保存时仍走现有绑定表达式验证。

坐标动态绑定和路由中的检测结果谓词使用同一套标签生成器，避免一个位置中文、另一个位置仍显示原始表达式。绑定解析器和执行引擎不做修改。

## 用户可见文字巡检

巡检范围是 `flow_runner/ui` 中应用维护的用户可见字符串，以及被界面直接展示的已知应用错误：按钮、标签、页签、下拉选项、占位文字、工具提示、摘要、对话框、状态栏和诊断窗口。

处理规则：

- 能由 `capability_label()`、`field_label()`、`choice_label()` 或新的结果字段标签映射的内容统一走本地化层。
- 原来用内部键作为回退展示的地方，为所有当前注册值补齐中文映射。
- `AND`、`OR`、`NOT` 可作为解释性缩写保留在中文后；OCR、RGB、ID、JSON、Python、Tesseract、PaddleOCR-json 等名称保留。
- 文件路径、文件名、用户命名、变量名和自定义表达式不翻译。
- 已知的应用验证错误改为中文；无法控制的操作系统或第三方原始错误保留详情，并由中文上下文说明包裹，避免损失诊断信息。

测试中的英文内部值不属于界面文字，不做机械替换。

## 窗口宽高偏好

新增可注入 `QSettings` 的窗口偏好对象，使用独立键保存主窗口宽度和高度。该偏好属于本机，不写入 `Project.settings`，也不影响项目脏状态。

启动顺序为：

1. 读取已保存宽高。
2. 两个值都有效时优先使用，并分别限制到当前主屏幕可用宽高范围。
3. 缺少或无效时采用现有基于屏幕可用区域计算的默认尺寸。
4. 设置尺寸后将窗口放在当前屏幕可见区域内；本次只持久化宽高，不持久化位置。

正常关闭流程最终被接受时保存当前非最小化窗口尺寸。若窗口处于最大化状态，保存其正常几何尺寸，避免下次把最大化后的屏幕尺寸当作普通窗口尺寸。保存偏好不依赖项目保存是否成功，也不触发关闭确认。

## 错误处理与兼容性

- 启动文件自动填充失败时不清空用户已有配置；用中文提示说明无法确定的部分，允许用户手工填写。
- 绑定选项无法识别旧值时保留原表达式并进入自定义模式，禁止静默丢弃。
- 窗口偏好中的非数字、非正数或超大值按无效或屏幕边界值处理，不能阻止程序启动。
- 现有项目不需要迁移；动作配置、绑定字符串和三列宽度数据格式不变。
- 当前工作区中的 `data/project.json` 属于用户运行数据，不修改、不暂存、不提交。

## 测试策略

### 布局与动作归属

- 验证两个顶部工具栏不再加入主窗口。
- 验证每个现有动作只出现在指定列，按钮触发同一个 `QAction`。
- 调整三个分栏到窄、宽尺寸，验证响应式操作区行数增加和减少、控件不重叠。
- 保留启动、暂停、停止、撤销和上下文启用状态的现有行为测试。

### 步骤卡片

- 验证载入工作流后所有卡片正文可见且标题可见。
- 验证选择任意卡片只改变选中状态，其他卡片仍展开。
- 验证列表项高度覆盖完整摘要内容。

### 启动文件与摘要

- 对 `.py`、`.pyw`（存在与回退）、`.bat` 和普通可执行文件分别测试纯映射函数。
- 验证附加参数和自定义工作目录的保留规则。
- 验证文件对话框选择信号只对 `LaunchProcessConfig` 生效。
- 验证 Python、批处理、普通程序和录制文件摘要只显示目标文件名。

### 绑定与中文化

- 验证主要结果和命名子节点表达式到中文标签的映射与反向保存。
- 验证未知旧表达式进入自定义模式且原值不变。
- 验证动作和路由编辑器复用同一套中文绑定标签。
- 增加受控的用户可见字符串检查，允许约定的缩写和技术名称，避免扫描对象名、配置键或测试数据造成误报。

### 窗口偏好

- 使用临时 INI 格式 `QSettings` 验证宽高写入与下次实例恢复。
- 验证无记录时使用默认尺寸，非法值被忽略，超大值被限制在屏幕范围。
- 验证窗口尺寸保存不修改项目脏状态。

最后在 `QT_QPA_PLATFORM=offscreen` 下运行相关 UI 测试和完整测试集，再进行一次真实窗口手工验收：拖动三列观察换行、选择四类启动文件、检查步骤卡片和中文绑定、关闭后重启验证窗口宽高。

## 验收标准

1. 主窗口顶部不再拥挤，全部功能位于已确认的对应列下方。
2. 任一列宽变化时，其操作区自动增减行数，按钮保持可见且可用。
3. 所有步骤始终展开，卡片最上方始终显示步骤名称。
4. 四类启动文件按规则自动生成路径、参数和工作目录，并保留用户自定义内容。
5. 执行摘要显示目标文件名；普通绑定界面显示具体中文名称，内部数据保持兼容。
6. 应用维护的界面文字除约定缩写和技术名称外使用中文。
7. 窗口宽高自动保存并优先恢复，无值时才使用默认尺寸。
8. 自动化测试和真实窗口验收通过，现有项目与运行行为无回归。
