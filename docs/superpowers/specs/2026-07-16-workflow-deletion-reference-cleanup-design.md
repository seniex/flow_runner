# 流程删除引用清理设计

## 目标

修复流程被其它步骤路由引用时无法删除的问题。用户确认删除后，应用应在一次可撤销的项目变更中删除目标流程，并清理会因此失效的引用，同时继续保护仍被并行监控块依赖的流程。

## 根因

`MainWindow` 删除前只检查并行监控块依赖。`ProjectViewModel.remove_workflow()` 删除流程后会执行完整项目引用校验；如果其它步骤的路由目标或 `workflow_count` 谓词仍引用该流程，校验抛出 `ConfigurationError`。该异常没有被删除界面转换为可操作提示，因此表现为确认后流程仍然存在。

## 删除行为

删除目标流程时执行以下原子变更：

1. 如果任一并行监控块包含目标流程，继续阻止删除，并显示涉及的并行块名称。并行块成员不得被静默修改。
2. 扫描其它流程的全部步骤，删除以下完整路由规则：
   - 路由目标为待删除流程的 `jump_workflow` 或 `call_workflow`；
   - 路由谓词为 `workflow_count`，且计数对象为待删除流程。
3. 删除目标流程本身。
4. 如果目标流程是 `settings.entry_workflow_id`，将入口设置为删除后项目顺序中的第一个流程；若项目已无流程，则移除该设置。
5. 对清理后的完整项目执行现有引用校验，通过后只提交一次历史记录，使一次撤销能够恢复流程、路由和入口设置。

删除整个路由规则而不是仅移除谓词，避免原本有条件的路由意外变成无条件路由。目标流程自身包含的路由随流程一起删除，不需要单独处理。

## 用户界面

确认框应说明删除流程会同时移除指向它的路由规则，并显示受影响路由数量。并行块依赖仍在确认框之前拦截。

删除成功后：

- 流程树不再显示目标流程；
- 步骤和属性面板清空已删除流程的选择；
- 状态栏报告已删除流程及清理的路由数量；
- 保存状态和撤销状态按现有机制刷新。

## 代码边界

- `ProjectViewModel` 负责查找引用、构造清理后的不可变项目并原子提交，业务规则不放入 Qt 控件。
- `MainWindow` 只负责并行块拦截、确认文案、调用删除操作和显示结果。
- 不修改项目持久化格式，不修改路由或流程 UUID 语义，不扩展到流程组删除行为。

## 错误处理

- 不存在的流程 ID 继续抛出 `KeyError`。
- 并行块依赖继续抛出或显示现有 `ConfigurationError` 语义。
- 清理后若仍有其它无效引用，现有项目校验继续拒绝提交，原项目和撤销栈保持不变。

## 测试

采用测试驱动方式覆盖：

1. 被跳转/调用路由引用的流程可以删除，并删除对应路由规则。
2. 被 `workflow_count` 谓词引用的流程可以删除，并删除对应完整路由规则。
3. 删除入口流程后自动选择第一个剩余流程；删除最后一个流程后移除入口设置。
4. 一次撤销完整恢复流程、路由和入口设置。
5. 并行块依赖仍阻止删除且不出现确认框。
6. UI 确认文案包含受影响路由数量，删除后选择和面板状态正确。
7. 无引用流程的现有删除行为保持不变。

## 非目标

- 不自动编辑并行监控块。
- 不保留指向已删除流程的失效路由。
- 不自动重定向路由到其它流程。
- 不在本次修复中改变流程组删除语义。
