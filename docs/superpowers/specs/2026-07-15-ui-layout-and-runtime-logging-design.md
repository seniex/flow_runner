# Flow Runner UI 布局与运行日志改进设计

日期：2026-07-15
状态：已确认，等待书面规格复核

## 目标

在不改变工作流业务语义和现有项目模型的前提下，修正本轮真实验收发现的五项 UI
问题，并为明确的 `system.wait` 等待动作增加每秒更新的倒计时：

1. 保存并恢复“组 / 步骤 / 配置”三列宽度。
2. 消除步骤列表文字重叠，按选中状态切换标题和说明。
3. 让顶部工具栏按钮的可点击、悬停、按下和禁用状态清晰可辨。
4. 让窗口操作的三个常用配置保持单行并按操作直接切换。
5. 将运行日志拆成默认中文简洁模式和可选调试模式。
6. 仅对明确的等待动作显示每秒更新、可暂停和可取消的倒计时。

根目录 `project.json`、验收产生的 `templates/` PNG、`flowUI.png`、`BGUI.png` 和备份文件
均属于用户数据或验收证据，本轮不得覆盖、回退或清理。

## 非目标

- 不为检测轮询间隔、动作重试间隔、资源等待或普通延迟显示倒计时。
- 不改变 `AutomationStep`、路由或能力配置的持久化格式。
- 不把视觉颜色写入 Python；所有视觉状态继续由正式 QSS 管理。
- 不删除已有 `logs/runtime.jsonl` 或其它历史日志，也不迁移或重写历史日志。
- 不在本轮增加日志搜索、导出、上传或远程收集。

## 三列布局和宽度持久化

主工作区改为两层直接结构：

- 外层垂直分割器：上方工作区、下方运行日志。
- 上方单个水平分割器：流程组导航、步骤卡片、详细配置三列。

这样三列宽度由同一个 `QSplitter` 管理，不再通过两个嵌套水平分割器间接计算。
水平分割器保存为 `MainWindow.workspace_splitter`，便于恢复和测试。

项目设置增加：

```json
{
  "settings": {
    "ui_layout": {
      "column_widths": [240, 420, 780]
    }
  }
}
```

行为规则：

- 启动时读取三个正整数；字段缺失、数量错误或包含非正数时使用默认比例。
- 在窗口完成初始尺寸设置后恢复宽度，避免 Qt 在窗口尚未布局时覆盖设置。
- 拖动分割器只记录待保存宽度并让窗口进入修改状态，不连续写入 ViewModel 历史。
- 点击保存或选择“保存并关闭”时，将当前宽度一次性合并到项目设置后再原子保存。
- 撤销优先丢弃尚未保存的布局宽度并恢复最近保存值，然后才进入项目历史。
- “不保存并关闭”不写入宽度。
- 本轮只持久化三列宽度；日志区域高度继续保持运行时可拖动但不写入项目配置。

## 步骤卡片状态

文字重叠的根因是 `QListWidgetItem` 委托和 `StepCardWidget` 同时绘制步骤名。修正后：

- 列表项不再绘制可见文本，只保存步骤 UUID 和辅助访问名称。
- 未选中卡片：显示步骤名，隐藏检测、执行、策略和路由说明。
- 选中卡片：隐藏步骤名，显示四类说明；动作仍保持一项一行。
- 卡片 QWidget 启用样式背景绘制，防止底层列表选中背景或文字透出。
- 卡片高度在状态切换后重新计算，保持列表滚动和选择行为正确。

## 顶部工具栏交互状态

顶部 `QAction` 实际由 `QToolButton` 呈现，因此正式 QSS 增加完整的 `QToolButton` 规则：

- 默认：有可见边框和轻微背景，与纯文本标签区分。
- 悬停：背景、边框和文字对比度提高。
- 按下：背景变深并保留边框反馈。
- 禁用：降低文字和边框亮度，不响应悬停样式。
- 选中或保持状态：使用独立强调背景。

启动、暂停和停止仍使用现有动作启用逻辑；本轮只改善视觉反馈，不改变运行控制语义。

## 窗口操作单行配置

`system.window_action` 的常用配置固定为“操作 / 窗口标题 / 窗口位置尺寸”。设计规则：

- 为该能力使用不换行的紧凑常用配置行。
- 操作下拉框和标题输入框使用受控宽度；四个几何数字框缩短并保持可编辑范围。
- 操作为激活、最小化或恢复时，隐藏几何字段。
- 操作为移动并调整大小时，在同一行立即显示几何字段。
- 高级 JSON 和 `WindowActionConfig` 验证保持不变；`move_resize` 仍必须提供几何值。
- 极窄窗口允许配置区出现水平滚动，不把三个字段重新拆成三行。

## 分级日志架构

### 默认中文简洁日志

默认界面和本次启动对应的 normal 日志文件使用同一个纯格式化器输出用户可读文本。每条
日志按适用情况包含：

- 本地时间；
- 任务启动、暂停、继续、停止、完成或失败；
- 流程组、流程和步骤的可读名称；
- 检测类型、关键目标、尝试次数和耗时；
- 动作摘要；
- 成功、未命中、超时、失败或取消结果；
- 选中的路由及可读目标名称；
- 中文错误原因和用于关联调试日志的错误编号。

默认模式不显示 UUID、帧 ID、场景代次、资源锁、原始 JSON、变量快照或完整异常详情，
也不逐条输出高频检测轮询。

日志格式化放入独立、Qt 无关的组件。它根据当前项目建立 UUID 到组、流程和步骤名称的
索引；项目编辑后重建索引。未知或已删除引用使用明确的“未知流程/步骤”占位，不抛出 UI
异常。

### 调试日志

项目设置增加顶层布尔值：

```json
{
  "settings": {
    "debug_logging": false
  }
}
```

设置对话框增加“启用调试日志”复选框，默认关闭。更改在下次启动时生效，与 OCR 引擎和
全局热键的启动期配置行为一致。

启用后：

- 界面在中文摘要后追加任务、流程、步骤和错误 UUID；
- 显示帧 ID、场景代次、资源等待、结果路由、重试信息和结构化详情；
- 本次启动的 debug 日志文件以每行一个完整 `RuntimeEvent` JSON 对象的形式写入全部字段。

关闭 debug 时：

- 本次启动只创建 normal 模式日志文件；
- 文件写入中文简洁内容，不写完整结构化事件；
- 已存在的 `logs/runtime.jsonl` 和历史模式日志保持原样，不删除或截断。

### 每次启动的日志文件

每次启动程序只为当前启用模式新建一个日志文件，目录固定为项目文件旁的 `logs/`：

```text
<项目名称>_<YYYYMMDD_HHMMSS>_normal.log
<项目名称>_<YYYYMMDD_HHMMSS>_debug.log
```

- 默认模式只创建 `normal.log`；启用调试后只创建 `debug.log`。
- 时间戳使用程序启动时的本地时间，同一次运行的全部任务写入同一个文件。
- 项目名称中的 Windows 非法文件名字符替换为下划线，去除结尾空格和句点；结果为空时使用
  `FlowRunner`。
- 若同一秒内已有同名文件，依次追加 `_2`、`_3`，禁止覆盖旧日志。
- normal 文件写中文简洁文本；debug 文件虽然扩展名为 `.log`，内容仍保持每行一个完整 JSON
  对象，便于人工检索和工具解析。
- 程序启动后切换设置不会更换当前文件；`debug_logging` 在下次启动时决定新文件模式。

### 界面日志行为

运行日志继续使用只读文本视图，设置最大保留行数并自动滚动到最新内容。独立的
`RuntimeLogController` 负责普通日志追加和活动等待管理；活动等待通过唯一等待 ID 绑定到
对应文本块，因此并行流程中的多个等待可以各自更新，不会覆盖其它日志。倒计时结束后该
文本块变为普通最终日志。

## 等待动作倒计时

采用“运行时边界事件 + UI 本地计时器”方案，不每秒写运行时事件。

运行时只为 `system.wait` 发送：

- `action.wait.started`：包含唯一等待 ID、总秒数、动作序号、流程 UUID 和步骤 UUID。
- `action.wait.finished`：等待正常完成。
- `action.wait.cancelled`：任务停止或取消导致等待结束。

界面收到开始事件后立即显示总剩余时间，并使用一秒计时器原位更新：

```text
[05:41:10] 等待中：等待加载，剩余 5 秒
[05:41:11] 等待中：等待加载，剩余 4 秒
[05:41:15] 等待完成：等待加载，共 5 秒
```

计时规则：

- 剩余秒数使用向上取整，避免刚开始就少显示一秒。
- `0` 秒等待直接显示完成，不启动计时器。
- 任务暂停时冻结活动倒计时；继续时从冻结值恢复。
- 停止或取消时立即停止计时器并显示“等待已取消”。
- 多个并行等待按任务 ID 与等待 ID 独立维护。
- 每秒 UI 更新不写入当前模式日志文件；文件只记录等待开始和最终状态。
- 若结束事件缺失但任务进入完成、失败或取消状态，界面兜底终止该任务的所有活动倒计时。

## 数据流和组件边界

1. Runner/StepExecutor 在准确的等待动作执行边界生成开始和最终事件。
2. RunnerBridge 继续通过现有 `eventReceived` 信号把事件送到 Qt 主线程。
3. `RuntimeLogFormatter` 将 RuntimeEvent 和项目名称索引转换为简洁或调试文本。
4. 独立 `RuntimeLogController` 负责追加普通文本、管理等待文本块和一秒计时器；
   `MainWindow` 只负责连接信号和刷新项目名称索引。
5. 应用组合层在启动时创建唯一模式文件：normal 使用简洁文本 sink，debug 使用完整事件
   JSON sink；两者不会同时创建。
6. 所有文件 sink 由 `SafeEventSink` 包装；写入异常被捕获并通过 RunnerBridge 的失败信号
   报告，不向 Runner 的自动化执行路径传播。

事件生成、文本格式化、Qt 倒计时和文件写入彼此分离，便于单独测试且不让日志展示逻辑
进入自动化业务能力。

## 错误处理和兼容性

- 所有新设置均为可选；旧项目在无设置时使用默认列宽和非调试日志。
- 无效列宽不阻止项目启动或保存，只回退默认值。
- normal 或 debug 模式日志写入失败由 `SafeEventSink` 捕获，通过现有失败信号报告，但不能
  中断自动化任务或改变 Runner 结果。
- 日志格式化器遇到未知事件类型时输出中文“运行事件”并保留事件类型，不丢弃事件。
- 当前验收写入的模板绝对路径、区域坐标、等待间隔和入口流程设置保持不变。

## 测试与验收

所有行为采用 TDD，先观察新增测试按预期失败，再实现最小改动。

自动化测试至少覆盖：

- 三列宽度只在保存时写入，重启恢复；无效设置回退；拖动不会产生连续项目历史。
- 卡片折叠只显示标题，展开只显示说明，列表委托不再绘制重复文字。
- 正式 QSS 包含 `QToolButton` 的悬停、按下、禁用和选中状态。
- 窗口操作三个常用字段在宽度足够和常规验收宽度下保持同一行，几何字段按操作切换。
- 简洁日志使用中文名称、结果、路由和错误；调试日志包含完整诊断字段。
- 设置对话框往返 `debug_logging`；应用每次启动创建项目名、时间戳和模式组成的新文件，
  同秒重名不覆盖。
- 等待开始、正常完成、暂停/继续、取消、零秒等待和多个并行等待。
- UI 倒计时使用可控时钟或手动触发定时器，不依赖真实长时间睡眠。
- 完整 UI 套件、全量 pytest、Ruff、格式、严格 mypy、compileall 和 diff 检查。

真实验收：

1. 调整三列宽度，保存并重启，宽度保持。
2. 切换步骤，确认标题和说明不重叠且按选中状态互斥。
3. 悬停、按下和禁用顶部按钮时视觉状态可辨。
4. 窗口操作三个配置保持单行，几何字段随操作直接显示或隐藏。
5. 默认日志可直接阅读且生成新的 normal 文件；启用调试并重启后出现完整诊断字段并生成
   新的 debug 文件。
6. 运行含等待动作的步骤，倒计时每秒原位更新；暂停、继续和停止行为正确。
