# Flow Runner 计划B：运行数据集中与项目整理设计

日期：2026-07-15
状态：书面规格已确认

## 目标

计划B完成本轮验收遗留修正，并将项目从“源码、业务配置、运行产物和历史资料混放”整理为
边界清晰、可继续维护的状态：

1. 修复日志文件名、取消检测次数，以及流程组、流程和步骤的独立显示序号。
2. 将活动配置、配置备份、识别截图、录制和日志集中到根目录固定的 `data/`。
3. 安全迁移现有真实业务数据并更新所有受影响的资源引用。
4. 归档仍有追溯价值的旧资料，只删除已经确认无运行引用且可由 Git 恢复的历史实现。
5. 清理过期状态文档，记录全局热键通过，并将多显示器和 Tesseract 实机验收延期。
6. 生成根目录 `PLAN_B.md`，让新会话能够从当前工作树继续实施而不重复已完成工作。

## 明确延期项目

以下项目不属于计划B的实施或完成条件：

- 多显示器实机验收：标记为 `DEFERRED`，等待具备第二块物理或虚拟显示器时恢复。
- Tesseract 实机验收：标记为 `DEFERRED`，等待用户决定安装程序、语言包和 Python 依赖时恢复。

延期不是自动化测试通过，也不是永久取消。状态文档必须保留恢复验收所需的环境条件和步骤，
但不得继续把它们描述为计划B的阻塞项。

## 非目标

- 不修改工作流的业务语义、UUID、执行顺序或路由含义。
- 不重新转换旧项目，不重建当前 `project.json`。
- 不自动安装 Tesseract、语言包或其它第三方程序。
- 不执行多显示器、真实鼠标、真实键盘或游戏窗口操作。
- 不删除 PaddleOCR-json、`.git`、`.worktrees` 或来源不明确的用户文件。
- 不长期维护根目录旧布局和 `data/` 新布局两套活动配置。
- 不自动提交、合并、推送、打标签或发布。

## 推荐方案与取舍

采用“一次性显式迁移到固定 `data/` 目录”的方案。

没有选择启动时自动迁移，因为应用启动不应在用户未确认时隐式移动真实配置。没有选择长期同时
支持新旧路径，因为双路径发现、保存位置和备份规则会长期增加歧义。计划B实施期间在完成清单、
哈希和验证后执行一次迁移；迁移完成后，默认应用入口只使用 `data/project.json`。

测试和显式传入 `project_path` 的调用继续允许使用任意临时路径，保证组件可测试，也保留将来打开
其它项目的能力。固定 `data/` 仅约束默认项目和本仓库的真实运行数据布局。

## 目标目录结构

```text
data/
├─ project.json
├─ backups/
│  └─ project.<时间戳>.bak.json
├─ templates/
│  ├─ template_<时间戳>.png
│  └─ legacy/
├─ recordings/
│  ├─ latest.json
│  └─ legacy/
├─ logs/
│  ├─ flow_runner_<时间戳>_normal.log
│  └─ flow_runner_<时间戳>_debug.log
└─ legacy/
   ├─ config/
   └─ scripts/
```

目录职责如下：

- `project.json`：唯一活动业务配置。
- `backups/`：应用保存产生的最多五份配置备份。
- `templates/`：框选截图及仍被活动项目引用的识别图片。
- `recordings/`：当前录制和仍被活动项目引用的旧录制。
- `logs/`：每次启动产生的 normal 或 debug 日志。
- `legacy/`：只用于追溯、不再作为活动配置直接读取的旧转换源和旧脚本。

第三方 PaddleOCR-json 不属于配置或运行产物，继续位于现有忽略目录。设计参考图属于文档资产，
迁移到 `docs/assets/ui-references/`，不放入 `data/`。

## 统一路径组件

增加一个 Qt 无关的路径值对象，例如 `ApplicationPaths`，由仓库根目录或显式项目路径构造。
它至少提供：

- `project_file`
- `backup_directory`
- `template_directory`
- `recording_directory`
- `latest_recording_file`
- `log_directory`
- `legacy_directory`

应用组合层只构造一次该对象，并把明确路径传给持久化、截图、录制和日志组件。各组件不得继续
自行使用 `path.parent / "logs"`、`path.parent / "recordings"` 或
`project_directory / "templates"` 推断运行目录。

`create_application(project_path=...)` 的规则：

- 未传入：使用当前工作目录下的 `data/project.json` 和同一 `data/` 树。
- 显式传入：该文件本身仍是项目配置；默认辅助目录从其父目录派生，测试可以继续在 `tmp_path`
  中隔离运行。

## 配置保存与备份

`ProjectStore` 接受可选的独立备份目录。保存事务仍保持：

1. 在活动配置旁写临时文件。
2. 刷盘并重新加载验证。
3. 将旧活动配置复制到 `data/backups/`。
4. 原子替换活动配置。
5. 仅保留最新五份备份。

备份文件名继续包含纳秒时间戳并保留 `.bak.json` 形式。备份清理只允许作用于明确的
`backup_directory`，不能扫描或删除根目录其它 JSON。

## 日志遗留修正

### 文件名

日志的“项目名称”解释为应用/仓库目录名，而不是 `Project.name` 内部显示名称。当前默认项目的
文件名固定为：

```text
flow_runner_<YYYYMMDD_HHMMSS>_normal.log
flow_runner_<YYYYMMDD_HHMMSS>_debug.log
```

显式打开其它项目路径时，使用该项目数据目录的名称；名称仍执行 Windows 非法字符清理和同秒
防覆盖处理。

### 取消检测次数

条件步骤在取消前已经完成的每次条件评估都必须计入 `condition_attempts`。取消可能发生在：

- 条件评估前：次数为 0。
- 某次无匹配后的间隔等待：保留已经完成的次数。
- 条件命中后的动作执行：保留命中所用次数。
- before/after hook 或资源等待中：保留取消前已经完成的条件次数。

不得通过读取格式化日志反推次数。`StepExecutor` 必须在产生取消 `StepResult` 时直接保留真实计数，
Runner、normal 格式化器和 debug JSON 共同使用该结果。

### debug 日志

debug 文件继续保持“一行一个完整 `RuntimeEvent` JSON”。本轮真实文件已经验证事件模型、嵌套
`StepResult`、路由、等待配对和任务内时间顺序均有效；计划B只修正取消结果中的检测次数，不改变
debug 文件协议。

## 流程组、流程与步骤显示序号

三级序号都是纯展示信息，使用两位起始编号，但各自在自己的直接容器内重新从 `01` 开始：

```text
01. 不思议挂机A
  01. 开始游戏
    01. 检测开始按钮
    02. 点击开始按钮
  02. 领取奖励
02. 不思议挂机B
```

规则：

- 流程组顺序来自 `project.groups`。
- 流程序号只取该流程在所属 `group.workflows` 中的位置，并在每个流程组内独立编号。
- 步骤序号只取该步骤在所属 `workflow.steps` 中的位置，并在每个流程内独立编号。
- 不把三级序号拼成 `1.2.1`、`01-02-03` 或其它层级编号。
- 新增、复制、删除、移动或发生重排后，相应直接容器内的序号自动重新计算。
- 不按名称重新排序，不修改 `FlowGroup.name`、`Workflow.name`、`AutomationStep.name`、UUID、
  项目 JSON 或路由引用。
- 左侧流程树显示带序号的流程组和流程；顶部启动选择框显示带序号的流程组和流程。
- 步骤卡片始终保留独立序号标签。未选中时显示“序号 + 步骤名”；选中时步骤名按现有规则隐藏，
  但序号仍可见，旁边显示检测、动作、策略和路由摘要。
- 任何用于选择流程组、流程或步骤的可读下拉框、列表和路由目标摘要都使用各自的独立序号。
- normal 中文日志和 debug 界面中文摘要的位置格式为
  `01. 流程组 / 01. 流程 / 01. 步骤`，三个编号各自独立计算。
- debug 文件中的原始事件继续以 UUID 作为稳定关联键，不把展示序号写入领域模型。
- 重命名输入框和项目 JSON 只显示、保存原始名称，不把序号当作名称的一部分。

序号格式化和 UUID 到三级位置的索引应集中在纯函数或小型索引组件，避免 UI、编辑器和日志分别
实现不同规则。

## 一次性迁移事务

迁移前必须确认 Flow Runner 未运行，并记录：

- 所有待迁移文件的绝对路径、大小和 SHA-256。
- 当前 `project.json` 的模型校验和引用校验结果。
- 当前 Git 状态，明确区分用户数据、未提交实现和待清理历史文件。

迁移按 staging 方式执行：

1. 创建临时 staging 目录，不直接覆盖目标文件。
2. 复制并验证活动项目、备份、日志、截图和录制。
3. 根据迁移映射更新项目中的相对或绝对资源路径。
4. 对迁移后的项目执行 Pydantic、UUID 引用和外部文件存在性校验。
5. 将 staging 中的完整 `data/` 原子或逐项安全落位。
6. 重新从 `data/project.json` 加载并复核哈希/模型差异。
7. 只有新布局验证通过后，才移除或归档旧位置文件。

如果任一步失败：

- 不删除旧文件。
- 删除不完整的 staging，而不是删除已验证的用户数据。
- 报告具体失败路径和校验项。
- 不尝试用空项目或旧转换源覆盖当前业务配置。

## 项目清理策略

采用“先归档、只删除确认无引用项”。

### 可以删除

满足以下全部条件后，从当前工作树删除：

- 当前源码、测试、文档和项目配置均无引用。
- 属于已被新包实现取代的历史实现。
- 已存在于 Git 历史，可恢复。

当前候选为 `flow_runner_p1.py`、`flow_runner_p2.py` 和 `flow_runner_p3.py`。实施时仍需再次运行
全仓库引用搜索后才能删除。

### 应归档

- 已完成的根目录设计、实施计划和修复计划迁入 `docs/archive/`。
- 改进A完成状态和真实验收结论保留，但不继续作为活动执行计划。
- `flowUI.png`、`BGUI.png` 迁入 `docs/assets/ui-references/`。
- 旧转换报告与迁移源应保持相互可追溯。

### 可以清理的缓存

`.pytest_cache/`、`.mypy_cache/` 和 `.ruff_cache/` 可删除并由工具重新生成。`.superpowers/`、
`.worktrees/` 和未知隐藏目录不得作为普通缓存删除。

## 状态文档收束

`REFACTOR_STATUS.md` 改为短而当前的状态摘要，至少包含：

- 当前分支和未提交计划B前置实现概况。
- 已完成的核心功能和最后一次真实自动化基线。
- 用户确认的 UI 1–4、等待倒计时和全局热键验收结果。
- 日志文件名、取消检测次数以及流程组、流程、步骤独立序号仍属于计划B待办。
- 多显示器与 Tesseract 为 `DEFERRED`。
- 新运行数据目录为 `data/`。

`REAL_ENVIRONMENT_CHECKLIST.md` 增加 `DEFERRED` 状态说明，并把全局热键更新为用户实测
`PASS`：启动、停止、暂停/继续、录制/停止录制均成功。不得把用户未执行的多显示器或
Tesseract 测试写成通过。

旧计划保留最终结果后归档，根目录只保留 README、当前状态、真实环境清单和 `PLAN_B.md` 等
仍对后续工作有直接价值的入口文档。

## 测试策略

全部行为按 TDD 实施，至少覆盖：

1. 默认路径为 `data/project.json`。
2. 显式 `project_path` 仍可在临时目录独立运行。
3. 配置备份写入独立 `backups/` 并只保留五份。
4. 截图、录制和日志写入各自 `data/` 子目录。
5. 日志文件名使用 `flow_runner`，同秒不覆盖。
6. 条件检测一次或多次后取消，结果保留真实次数；首次检测前取消仍为 0。
7. 流程组、流程和步骤在各自直接容器内独立编号，增删复制移动后刷新且不修改原始名称。
8. 迁移映射更新录制和模板路径，迁移后所有外部文件存在。
9. 迁移失败保留旧文件，不产生半迁移活动配置。
10. debug 文件仍能逐行通过 `RuntimeEvent.model_validate_json()`。

验证顺序：专项单元测试、相关 UI 测试、全量 pytest、Ruff、格式检查、严格 mypy、
`compileall`、`pip check` 和 `git diff --check`。

## 手工验收

计划B自动化验证完成后，只需要用户确认：

1. 从默认启动方式打开的是 `data/project.json`。
2. 原有流程组、流程、步骤和资源仍能正常加载。
3. 流程组、流程和步骤分别显示 `01.`、`02.` 等独立自动序号，不出现 `1.2.1` 层级编号。
4. 停止未命中的持续检测时，日志显示真实检测次数。
5. 新日志、截图和录制只出现在 `data/` 对应目录。
6. 根目录不再出现新的项目备份、日志、截图或录制文件。

上述验收不包含延期的多显示器和 Tesseract。
