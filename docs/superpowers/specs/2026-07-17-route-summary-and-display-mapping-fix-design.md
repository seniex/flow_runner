# 路由摘要完善与显示器映射修复设计

## 目标

本次工作包含两个相互独立但都影响编辑体验的改进：

1. 完善步骤卡片中的路由摘要，使用户无需打开属性面板即可看清每条路由的结果、附加条件和完整目标位置。
2. 修复框选区域、框选截图和点选坐标因 Qt 屏幕名与 Win32 设备名不同而无法建立物理坐标映射的问题。

两项改动都不得改变项目 JSON、UUID、路由执行语义或用户现有业务配置。

## 当前状态与根因

### 路由摘要

`StepCardWidget` 当前仅用 `结果→目标类型` 表示路由，多条路由使用分号拼在一行。它没有接收完整项目上下文，因此无法解析目标流程组、流程、步骤和展示序号，也没有显示路由谓词。

`RouteEditor` 已经能够显示完整的流程/步骤路径、执行次数谓词、比较符号和“否则”语义，但这些格式化逻辑目前是编辑器私有实现，步骤卡片无法复用。

### 显示器映射

`display_mappings_for_frame()` 当前要求 `QScreen.name()` 与 Win32 `MONITORINFOEX.szDevice` 忽略大小写后完全相等。本机实测：

- Qt 屏幕名称：`27E1Q`
- Qt 几何：`(0, 0, 2560, 1440)`
- Win32 设备名称：`\\.\DISPLAY1`
- Win32 物理矩形：`(0, 0, 2560, 1440)`

两套 API 描述的是同一显示器，但一个返回显示器型号，一个返回 GDI 设备名，因此名称精确匹配失败并产生“无法匹配显示器物理坐标：27E1Q”。

## 路由摘要设计

### 共享格式化边界

将路由摘要整理为 Qt 无关的纯展示逻辑，由步骤卡片和路由编辑器共同调用。格式化器接收：

- 当前项目的 `ProjectDisplayIndex`；
- 当前步骤及其有序路由列表；
- 当前条件结果绑定的中文标签映射。

格式化器只生成字符串，不读取控件状态，不修改领域对象。

### 单条路由格式

每条路由独占一行，并保留配置顺序：

```text
路由 1：成功 → 跳转到 01. 不思议挂机A / 03. 游戏加载完毕 / 01. 游戏加载完毕
路由 2：失败 且 01. 不思议挂机A / 02. 等待游戏加载 / 01. 固定等待执行次数 ≥ 3 → 结束任务
```

显示内容包括：

- 路由结果：成功、未匹配、超时、失败或取消；
- 同结果的有条件路由之后存在无条件路由时，保留“否则”说明；
- `workflow_count` 和 `step_count` 使用带序号的完整项目路径及“执行次数”；
- 任务变量、流程变量和结果绑定谓词显示中文名称、比较符号与期望值；
- `next_step` 显示目标步骤的“组 / 流程 / 步骤”完整路径；
- `jump_workflow` 和 `call_workflow` 显示目标“组 / 流程”，并在目标流程非空时追加首个步骤；
- `return` 和 `end` 使用现有中文标签。

未知或已失效引用继续显示明确的“未知流程/步骤”，不在展示阶段抛出异常。

### 多路由和无路由

步骤卡片使用一个支持换行和自动换行的摘要标签，每条显式路由以换行符分隔，不使用分号拼接。

无显式路由时，不再显示容易误解的“路由：无”，而显示当前执行器的真实隐式语义：

```text
路由：成功时顺序进入下一步骤；其它结果结束
```

这只是展示说明，不创建实体路由，也不改变最后一步成功后结束当前任务的行为。

### 项目上下文传递

`StepListPanel` 保存当前项目并构建 `ProjectDisplayIndex`。主窗口项目变更时同步刷新该上下文；设置工作流时，步骤卡片获得项目索引，用于解析跨流程目标和谓词引用。

## 显示器映射设计

### 显示器身份信息

扩展物理显示器信息，使每个 `PhysicalDisplay` 除 GDI 设备名和物理矩形外，还可携带 Windows 提供的显示器型号或友好名称别名。现有设备名仍保留，避免破坏测试和调用方。

Windows 枚举层通过 `EnumDisplayDevicesW` 取得显示器设备 ID，再读取对应注册表 EDID 的
显示器名称描述符，建立：

```text
\\.\DISPLAY1 ↔ 27E1Q ↔ 物理矩形
```

UI 映射层不直接调用额外 Win32 API。

### 分层匹配算法

为每个 Qt 屏幕按以下顺序寻找物理显示器：

1. 设备名或任一别名忽略大小写后完全匹配；
2. 若名称不匹配，按 Qt 几何、设备像素比和物理矩形筛选几何兼容候选；
3. 仅有一个候选时使用该候选；
4. 没有候选或存在多个候选时拒绝猜测，并报告 Qt 名称、逻辑矩形、DPI 比例及物理候选信息。

一个物理显示器只能匹配一个 Qt 屏幕，防止多个屏幕误用同一物理矩形。

### 几何兼容

几何回退比较以下信息：

- Qt 屏幕逻辑宽高乘以 `devicePixelRatio()` 后与物理宽高一致；
- 物理矩形与捕获画面存在合理交集；
- 多显示器场景中优先保持屏幕相对布局和原点关系。

名称别名是主要修复路径；几何匹配是驱动、远程桌面或虚拟显示器没有稳定名称时的安全回退，不按枚举顺序配对。

### 影响范围

`RegionCaptureService`、模板截图和 `PointCaptureService` 都通过 `display_mappings_for_frame()` 建立覆盖层映射，因此共享一次底层修复。窗口目标和完整桌面目标继续使用相同坐标变换。

### 隐藏窗口的捕获排除

仅调用 `QWidget.hide()` 后立刻截图仍可能捕获到 Windows DWM 隐藏动画中的完整或半透明残影。
启用“框选时隐藏程序界面”后，选择会话先对当前进程的可见顶层窗口设置
`WDA_EXCLUDEFROMCAPTURE`，再隐藏窗口、处理 Qt 事件并调用 `DwmFlush()`，之后才采集画面。
选择完成、取消或异常时，先恢复原显示亲和性，再恢复原来可见的窗口。

## 错误处理

- 显示器别名获取失败时保留设备名和几何回退，不阻止枚举其它显示器。
- EDID 缺失、注册表不可读或显示器没有名称描述符时返回空别名，不阻止捕获。
- `Get/SetWindowDisplayAffinity` 或 `DwmFlush()` 不可用时安全回退到普通隐藏逻辑，并始终恢复窗口。
- 无法唯一匹配时继续阻止打开覆盖层，避免记录错误坐标。
- 错误信息保留“框选区域失败”“框选并截图失败”或“点选坐标失败”的操作上下文，并追加可诊断的屏幕信息。
- 路由摘要遇到未知引用时使用占位名称，不影响步骤卡片构建。

## 测试

所有行为采用 TDD，先观察新增测试按预期失败。

### 路由摘要

- 跳转步骤显示带序号的组、流程和步骤路径；
- 跳转/调用流程显示组、流程和首步骤；
- 流程/步骤执行次数谓词显示完整路径、比较符号和期望次数；
- 结果绑定和变量谓词保留中文摘要；
- 同结果条件路由后的无条件路由显示“否则”；
- 多路由逐行显示；
- 无路由显示隐式成功/非成功行为；
- 项目重排后摘要序号随 `ProjectDisplayIndex` 更新，UUID 和 JSON 不变。

### 显示器映射

- Qt 名称 `27E1Q` 能通过物理显示器别名匹配 `\\.\DISPLAY1`；
- 名称匹配忽略大小写；
- 单显示器名称不一致时可通过几何和 DPI 唯一回退；
- 150% DPI、负坐标副屏和跨屏窗口保持现有换算结果；
- 相同尺寸的多个候选无法唯一判断时明确失败；
- 一个物理显示器不会被重复分配；
- 区域和点选服务能够使用修复后的映射启动覆盖层。
- 排除捕获、隐藏、DWM 刷新、截图和亲和性恢复按固定顺序执行；异常和取消也恢复状态。

## 文档与验收

README 补充步骤卡片完整路由摘要和显示器名称兼容说明。自动化验证后进行真实 GUI 验收：

1. 查看包含多条路由和计数条件的步骤卡片；
2. 在当前 `27E1Q` 显示器上分别执行框选区域、框选截图和点选坐标；
3. 验证桌面绝对坐标、窗口相对坐标及保存后的值正确；
4. 不运行会操作真实游戏窗口的自动化流程。

## 非目标

- 不新增运行期间累计的成功/失败统计。
- 不修改路由匹配顺序、比较语义或默认顺序执行行为。
- 不改变显示编号规则或把编号写入项目数据。
- 不按显示器枚举顺序进行不可靠配对。
- 不修改用户现有 `data/project.json`。
